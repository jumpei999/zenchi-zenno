# ライセンス戦略

OSS ライセンスおよび将来の商用リポジトリの Decision レコード。

**ステータス:** フェーズ 0 ～ 1 について決定。 **フェーズ 2 (チーム / クラウド)** の実装前に再確認してください。
**関連:** [commercial-boundary.md](commercial-boundary.ja.md) · [ライセンス](../LICENSE) · [GOVERNANCE.md](../GOVERNANCE.ja.md)

---

<a id="decision-phase-01"></a>

## Decision (フェーズ 0 ～ 1)

| アイテム                  | Decision                                                                                                        |
| ------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **現在の OSS ライセンス** | **MIT** (保持)                                                                                                  |
| **理論的根拠**            | アーキテクチャとパーソナル MVP の出荷中に、コミュニティの早期導入とコントリビュータのスムーズさを最大限に高める |
| **商法**                  | このリポジトリにはまだありません。フェーズ 1 には収益機能はありません                                           |
| **ゲートを再訪**          | チーム Workspace、マネージド同期 SaaS、またはエンタープライズ ポリシーを実装する前に                            |

<a id="decision-for-phase-2-pre-commitment"></a>

## Decision フェーズ 2+ (事前コミットメント)

チーム/クラウドの作業が開始されると、RFC によって変更されない限り、次の **デフォルト プラン** を採用します。

| アイテム                                          | 計画された選択                                                                 | なぜ                                                                                                      |
| ------------------------------------------------- | ------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| **OSS ライセンスの移行**                          | **Apache ライセンス 2.0**                                                      | エンタープライズ向け。特許付与;クラウドネイティブ OSS で一般的。企業向けの MIT よりも明確な貢献ストーリー |
| **SaaS フリーローディングが深刻な場合の代替方法** | **AGPL-3.0** (ネットワークに接続された OSS サービス用)                         | OSS サーバーのクローズされた SaaS 再ホストに対するコピーレフトの強化                                      |
| **商業レポ**                                      | 個別の **`zenchi-zenno-cloud`**                                                | カーネルを開いたままにします。オープンコア境界をクリア                                                    |
| **商用ライセンス**                                | **BSL 1.1** (約 4 年後にライセンスを変更 → Apache) **または** プロプライエタリ | OSS カーネルのミッドフライトのライセンスを再取得することなく、コラボレーション/運用に課金する余地         |

**優先されるプライマリ パス:** Apache 2.0 (OSS) + BSL/proprietary (`zenchi-zenno-cloud`)。

**エスカレーション パス:** 資金豊富なホストが貢献せずに OSS API を再ホストする場合は、サーバー パッケージのみについて AGPL を再検討してください。RFC 経由で文書化してください。

<a id="what-stays-open-under-either-path"></a>

## どちらのパスでも開いたままになるもの

常に OSS ([commercial-boundary.md](commercial-boundary.ja.md) ごと):

- 正規の知識モデルとスキーマ
- Connector SPI および基本コネクタ
- ローカル個人 CLI
- 基本的な MCP 下りツール

決して「ペイウォール化されたカーネル オントロジー」を使用しないでください。

<a id="migration-checklist-execute-at-phase-2-start"></a>

## 移行チェックリスト (フェーズ 2 開始時に実行)

- [ ] コミュニティ通知 (ディスカッション + README) **LICENSE ファイル変更の 14 日以上前**
- [ ] `LICENSE` を Apache-2.0 テキスト (または RFC が優先する場合は AGPL) に置き換えます。
- [ ] README / パッケージのメタデータ SPDX 識別子を更新します
- [ ] Apache の場合、`NOTICE` を追加 2.0
- [ ] `zenchi-zenno-cloud` プライベートまたはパブリック BSL リポジトリを作成します
- [ ] ライブ クラウド リポジトリ URL でCommercial-boundary.mdを指定します。
- [ ] クラウド and/or エンタープライズ顧客のニーズについて、CLA と DCO を確認する
